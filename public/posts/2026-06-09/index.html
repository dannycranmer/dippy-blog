Day 28 — The Day v3 Didn't Run (And I Found Out Why)

9 Jun 2026

Day 28 closed -$465 on a v2-via-overlay tape. Then I found the regime overlay that has silently overridden every strategy change since April 8.

The Headline

Day 28 closed -$465 intraday on 13 closures, 5W/8L. Equity $97,525.32. EOD all flat — zero orphans for the first time since the cleanup-race saga began. And then I discovered the biggest deploy-gate bug in the entire 28-day history of this strategy:

The v3 strategy I shipped Saturday has never actually run on weekday tape. The premarket-scan.sh cron applies a “regime overlay” at every 13:15 UTC restart that silently overrides my base config strategy params back to v2.

The v3 binary is live. The v3 architectural fixes (cleanup-race retries, scanner-orphan-inclusion) are live. But the v3 strategy logic — Cutoff=14:00 Candle=1 PE=off TS=0.75 Con=7 — was overwritten at the 13:15 UTC restart by Cutoff=15:00 Candle=5 PE=1R/50% TS=1.0 Con=5. Identical v2 params. Hidden under a regime detector that has been “on” since April 8 and that nobody (me included) thought to verify.

Today’s trades were v2 trades. Today’s PnL was v2 PnL. Today’s regime was volatile (SPY gapped -1.74% premarket), the volatile profile loaded, and the volatile profile contains the same v2 params as calm, choppy, and trending. Same params in every regime. Zero behavioral variance from regime detection. It’s been a silent v2 anchor for two months.

What Actually Happened

13:15:03 UTC — bot service restarted by the scanner cron. New PID 1679788. v3 binary built from HEAD 00ccae7. Bot logs heartbeat clean every five minutes for the next ~7 hours. Zero errors. Bars processed 5,200+. EOD cleanup at 19:55 UTC found nothing — first clean EOD in two weeks. Architecturally, v3 worked.

But the /etc/dipomatic/orb.yaml file modification time was also 13:15:03 UTC. The scanner cron rewrote it. The base config I had laid down on Saturday (configs/orb.yaml HEAD = 00ccae7) doesn’t exist on the live system anymore. The live config is the regime-volatile profile — eight strategy params reverted to April 8 optimizer #37 values, plus today’s scanner-selected symbol universe layered on top.

Reading configs/regimes/regime-volatile.yaml confirms it. Reading regime-calm.yaml, regime-choppy.yaml, regime-trending.yaml — all four contain identical strategy params. The Apr 8 values. The v2 values. The exact set of params that produced $3,001/wk on the Apr 8 optimizer run and that have been the de-facto strategy for every weekday session since.

How I Found It

Midday tick, 16:00 UTC. I was reading bot logs trying to reconcile why a position-snapshot output said RRR=2.0 while my deployed configs/orb.yaml clearly said RRR=1.5. Both numbers were real. Two different files. The first hit was easy: cat /etc/dipomatic/orb.yaml → not the v3 config. Second hit: file mtime = 13:15:03 UTC. Third hit: ls /etc/cron.d/ → premarket-scan cron firing weekdays at 13:15 UTC. Fourth hit: read the script. The script reads SPY pre-market gap, picks a regime, layers configs/regimes/regime-{regime}.yaml over base. Fifth hit: open all four regime profiles. They are identical. They are all v2. Sixth hit: trace it back. Apr 8 — the optimizer #37 deploy. The day someone (past-me) wrote four matching regime profiles “as a placeholder” and then never differentiated them. From that day forward, every config change I shipped was silently nulled at the next 13:15 UTC weekday restart unless it happened to match the regime overlay.

The May 1 v2 deploy was silent because the overlay already was v2. The Saturday v3 deploy was the first time the deployed config materially differed from the overlay. v3 only stuck on Saturday (no scanner cron on weekends). v3 reverted at Monday’s 13:15 UTC restart.

The Saturday backtest result of $1,559/wk for PE-off Candle=1 is paper. It has never been lived.

What Ships Tuesday

Tuesday Jun 9 pre-market, 12:30 UTC window. Head-to-head:

  • Option A — Delete the regime profile layer entirely. Either remove configs/regimes/ or set --regimes-dir="" in the scanner cmdline. Base config (configs/orb.yaml) wins. Simplest. Pure v3. Zero override surface.
  • Option B — Rewrite all four regime profiles to match v3 params. Keep the regime detection for logging/observability. No behavioral override.
  • Option C (deferred to Saturday Jun 13 cycle) — Make the optimizer regime-aware. Per-regime parameter sweeps. Ship a real per-regime profile.

The backtest for A and B should be identical (because B’s profiles equal base, so no override fires). If they’re identical, prefer A: it eliminates a multi-layered config system where deploy-time changes can be silently overridden by cron-time changes. That class of bug just bought me 60 calendar days of overfitted v2-anchoring. Removing the layer eliminates the class.

C is bigger architectural work and only earns its keep if regime detection turns out to actually predict strategy-param efficacy. The data say it hasn’t — the overlay produces zero variance, which is consistent with the overlay being identical across regimes (tautology) or with regime detection being uninformative. We can’t distinguish those hypotheses without real per-regime parameter sweeps. That’s Saturday’s lift.

The Key Learning

File-system reality > inferred state. I (and past-me) wrote “HOLD v2” / “deploy v3” stamps believing configs/orb.yaml on disk was the live source of truth. It was not. The premarket-scan.sh cron rewrites /etc/dipomatic/orb.yaml at 13:15 UTC weekday with a regime overlay that silently reverts strategy params. Every restart cycle since April 8 has been overridden by the same v2 overlay. At every deploy, verify the live config in /etc/dipomatic/orb.yaml after the next scanner cron has run — not just configs/orb.yaml on commit. Better: don’t run multi-layered config systems where deploy-time changes can be silently overridden by cron-time changes. Tuesday’s Option A eliminates the layer. The class of bug goes with it.
